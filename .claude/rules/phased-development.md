# 단계별 개발 순서 규칙

sdd.md와 prd.md 기반의 개발은 반드시 아래 4단계 순서를 따른다.
한 단계가 완료되기 전에 다음 단계로 넘어가지 않는다.

## 개발 단계 정의

### 1단계: 프로젝트 환경 초기화 및 구조 세팅
- 의존성 패키지 설치 (package.json, 설치 명령어)
- sdd.md에 정의된 폴더 구조 생성
- 기본 설정 파일 작성 (tsconfig, eslint, prettier, tailwind, next.config 등)
- .env.example 생성 (실제 시크릿은 절대 하드코딩 금지)
- Docker 관련 파일 (docker-compose.yml 등)
- 이 단계의 테스트: 빌드/린트가 에러 없이 통과하는지 확인

### 2단계: 데이터베이스 연동 및 공통 인터페이스/유틸리티 구현
- Prisma 스키마 작성 (sdd.md의 DB 모델링 기반)
- 마이그레이션 실행
- 공통 타입/인터페이스 정의
- 공통 유틸리티 (crypto, sanitizer, validator 등)
- 에러 처리 미들웨어 (RFC 7807)
- 인증/인가 미들웨어
- 이 단계의 테스트: DB 연결, 마이그레이션, 유틸리티 단위 테스트

### 3단계: 핵심 비즈니스 로직 및 백엔드 API 구현
- Service 레이어 (비즈니스 로직)
- Controller 레이어 (요청/응답 처리)
- Route 정의
- 위험 키워드 감지, 알림, 리포트 등 부가 서비스
- 이 단계의 테스트: API 엔드포인트별 단위/통합 테스트

### 4단계: 프론트엔드 주요 화면(UI) 컴포넌트 및 API 연동
- 페이지 컴포넌트 (App Router)
- 재사용 UI 컴포넌트
- Zustand 스토어
- API 서비스 레이어 (프론트)
- 커스텀 훅
- 이 단계의 테스트: 컴포넌트 렌더링 테스트, 통합 테스트

## 적용 규칙

- plan.md의 테스트 항목은 위 단계 순서에 맞게 배치되어야 한다
- tdd-cycle에서 "go" 실행 시, 현재 단계에 해당하는 테스트만 진행한다
- 각 단계가 끝나면 실행/테스트 방법을 안내하고 사용자 확인을 기다린다
- 사용자가 "다음 단계 진행해"라고 지시해야 다음 단계로 이동한다
- 코드 작성 시 최신 모범 사례(Best Practices) 적용, 재사용 가능한 모듈 분리
- 주석은 한국어로 친절하게 작성
- 중복 코드 방지를 위해 공통 모듈을 적극 활용
